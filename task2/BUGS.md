# Баг-репорты для API объявлений https://qa-internship.avito.com

## Серьезность: Critical

### ID: Bug-1
**Описание:** POST `/api/1/item` возвращает строку в поле `status` вместо объекта объявления со всеми полями запроса

**Серьезность:** Critical

**Окружение:** User-Agent: Chrome/143.0.0.0, Go 1.25.7 

**Шаги:**
1. Отправить POST `/api/1/item` с телом:
   ```json
   {
    "sellerID": 333333,
    "name": "Машина",
    "price": 10,
    "statistics": {
        "likes": 10,
        "viewCount": 10,
        "contacts": 10
    }
   }
   ```
2. Обратить внимание на код и тело ответа

**Фактический результат:** Статус код 200 OK.
Тело ответа включает в себя одно поле `status` с текстовым сообщением:
```json
{
    "status": "Сохранили объявление - 2b7699a8-151e-4090-89f3-71ebdbcb9752"
}
```

**Ожидаемый результат:** Статус код 200 OK.
Ответ представляет собой структуру объявления.
```json
{
        "createdAt": "2026-03-20 22:17:20.538996 +0300 +0300",
        "id": "2b7699a8-151e-4090-89f3-71ebdbcb9752",
        "name": "Машина",
        "price": 10,
        "sellerId": 333333,
        "statistics": {
            "contacts": 10,
            "likes": 10,
            "viewCount": 10
        }
    }
```

## Серьезность: High

### ID: Bug-2
**Описание:** POST `/api/1/item` отдает некорректную структуру ответа при ошибке создания объявления (`messages` пустое)  
**Примечание:** Воспроизводится на любой ошибке валидации полей (отсутствующие поля, невалидный тип, пустые значения и др).

**Серьезность:** High

**Окружение:** User-Agent: Chrome/143.0.0.0, Go 1.25.7

**Шаги:**
1. Отправить POST `/api/1/item` с телом (отсутствуют поля `name` и `sellerID`):
   ```json
   {
    "price": 10,
    "statistics": {
        "likes": 10,
        "viewCount": 10,
        "contacts": 10
    }
   }
   ```
2. Обратить внимание на код и тело ответа

**Фактический результат:** Статус код 400 Bad Request.
Тело ответа включает в себя поле `status`, `messages` пустое, сообщение лишь об одной ошибке в поле `message`:
```json
{
  "result": {
    "message": "поле name обязательно",
    "messages": {}
  },
  "status": "400"
}
```

**Ожидаемый результат:** Статус код 400 Bad Request.
Ответ представляет собой структуру, в которой есть объект `messages` с детализацией ошибок.
```json
{
  "result": {
    "message": "ошибка валидации",
    "messages": {
      "name" : "поле name обязательно",
      "sellerId" : "поле sellerId обязательно"
    }
  },
  "status": "400"
}
```

### ID: Bug-3
**Описание:** При отправке на POST `/api/1/item` отрицательных значений в полях `price`, `likes`, 
`viewCount` и `contacts` API успешно создаёт объявление (200 OK), хотя такие значения некорректны. 
Это нарушает бизнес-логику: цена и числовые показатели статистики не могут быть отрицательными

**Серьезность:** High

**Окружение:** User-Agent: Chrome/143.0.0.0, Go 1.25.7

**Шаги:**

1. Отправить POST `/api/1/item` с телом:
```json
{
  "sellerId": 333333,
  "name": "Машина",
  "price": -10,
  "statistics": {
    "likes": -10,
    "viewCount": -10,
    "contacts": -10
  }
}
```
2. Обратить внимание на код и тело ответа

**Фактический результат:** Код ответа 200 Ok.
Объявление успешно создается с отрицательными значениями

**Ожидаемый результат:** Код ответа 400 Bad Request.
В поле `messages` тела ответа указаны все поля с некорректными значениями.

### ID: Bug-4
**Описание:** POST `/api/1/item` при попытке создать объявление со значением 0 в полях `likes`, `viewCount`, `contacts` или `price` 
возвращает ошибку и не создает объявление. Хотя цена = 0, а также нулевая статистика на момент создания объявления 
обычно являются валидным сценарием

**Серьезность:** High

**Окружение:** User-Agent: Chrome/143.0.0.0, Go 1.25.7

**Шаги:**
1. Отправить POST `/api/1/item` с телом:
```json
{
  "sellerID": 333333,
  "name": "Машина",
  "price": 0,
  "statistics": {
    "likes": 0,
    "viewCount": 0,
    "contacts": 0
  }
}
```
2. Обратить внимание на код ответа

**Фактический результат:** Статус код 400 Bad Request с ошибкой на первом поле,
значение которого 0. Объявление не создается
```json
{
    "result": {
        "message": "поле price обязательно",
        "messages": {}
    },
    "status": "400"
}
```

**Ожидаемый результат:** Статус код 200 OK. Объявление успешно создается

    "message": "",

## Серьезность: Medium

### ID: Bug-11
**Описание:** При передаче специальных значений `../../../` и `../../../../../../` в path-параметры наблюдается различное поведение системы.
Один запрос блокируется внешней системой, другой самим API

**Серьезность:** Medium

**Окружение:** Postman 11.86.5

**Шаги:** 
1. Отправить GET-запрос `/api/1/item/../../../`
2. Обратить внимание на тело ответа
3. Отправить GET-запрос `/api/1/item/../../../../../../`
4. Обратить внимание на тело ответа

**Фактический результат:** Для `../../../`: возвращается JSON-ответ с ошибкой маршрута  
Для `../../../../../../`: возвращается HTML-страница с ошибкой

**Ожидаемый результат:** Все некорректные значения path-параметра обрабатываются единообразно.
API возвращает статус 400 Bad Request и тело ответа в формате JSON

## Серьезность: Low

### ID: Bug-12
**Описание:** Некорректная документация структуры ответа ошибки для эндпоинта GET `/api/1/statistic/:id` . 
Документация описывает структуру ответа при ошибках 404 как отличную от
других ошибок:
```json
{
  "result": "laborum",
  "status": "cillum enim eiusmod"
}
```
Однако фактически сервер возвращает ту же структуру ошибок, что и на других эндпоинтах.

**Серьезность:** Low

**Шаги:**
1. Отправить GET `/api/1/statistic/:id` с несуществующим id (7a133bff-1e14-4f33-8f71-66ea34025e28)
2. Открыть документацию к эндпоинту
3. Сравнить ответ api с документацией

**Фактический результат:** Документация не соответствует ответу api. Структура аналогична ошибкам на других эндпоинтах. 
Тело ответа:
```json
{
  "result": {
    "message": "statistic 7a133bff-1e14-4f33-8f71-66ea34025e28 not found",
    "messages": null
  },
  "status": "404"
}
   ```
**Ожидаемый результат:** Документация для 404 соответствует реализации, а также документации по другим ошибкам.  
Пример документации
```json
{
  "result": {
    "messages": {
      "culpa_b92": "<string>",
      "enim_24f": "<string>",
      "mollit_aa": "<string>"
    },
    "message": "<string>"
  },
  "status": "<string>"
}
```

### ID: Bug-13
**Описание:** Сообщения об ошибках на разных языках при GET `/api/1/item/:id`

**Серьезность:** Low

**Окружение:** Postman 11.86.5

**Шаги:**
1. Отправить GET `/api/1/item/:id` с рандомным несуществующим UUID (7a133bff-1e14-4f33-8f71-66ea34025e28)
2. Отправить GET `/api/1/item/:id` с произвольной строкой вместо UUID ("random_string")
3. Сравнить язык ошибок

**Фактический результат:** Ошибки различаются по языку(одна на русском, другая на английском), что делает систему неоднородной.  
Тело ответа GET `/api/1/item/:id` при 404 содержит 
``
"message": "item 7a133bff-1e14-4f33-8f71-66ea34025e28 not found" 
``  
Тело ответа GET `/api/1/item/:id` при 400 содержит 
``
"message": "ID айтема не UUID: random_string"
``

**Ожидаемый результат:** Все сообщения об ошибках валидации и поиска (400, 404) на русском языке
